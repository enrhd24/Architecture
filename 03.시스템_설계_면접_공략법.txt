시스템 설계 면접
: 두 명의 동료가 모호한 문제를 풀기 위해 협력
: 그 해결책을 찾아내는 과정에 대한 시뮬레이션

뭐하는 자리?
설계 기술을 시연하는 자리
설계 과정에서 내린 결정들에 대한 방어 능력
면접관의 피드백을 건설적인 방식으로 처리할 자질

목표
: 지원자가 협력에 적합한 사람
: 압박이 심한 상황도 잘 헤쳐 나갈 자질
: 모호한 문제를 건설적으로 해결할 능력이 있는지
: 설계의 순수성에 집착하여 타협적 결정을 도외시하는 경우


1단계. 문제 이해 및 설계 범위 확정
부정적 신호 : 요구사항이해하지 않고 답을 내놓는 행위
긍정적 신호 : 올바른 질문, 적절한 가정, 시스템 구축의 정보


2단계. 개략적인 설계안 제시 및 동의 구하기
긍정적 신호 : 개략적인 설계안을 제시하고 면접관의 동의를 얻는 것
- 지원자들과 대화하고 설계 과정에 개입하기를 즐긴다.
- 핵심 컴포넌트를 포함하는 다이어그램을 그려라 [클라이언트, API, 웹 서버, 데이터 저장소, 캐시, CDN, 메시지 큐]
- 시스템 규모에 관계된 제약사항들을 만족하는지 개략적으로 계산해보기

견고
- 시스템의 구체적 사용 사례
- 고려하지 못한 에지 케이스를 발견

ex> 피드발생 과 피드 생성
- 피드 발행 
: 사용자가 포스트를 올리면, 캐시/데이터베이스에 기록된다.
: 해당 사용자의 친구(friend) 뉴스 피드에 뜨게 된다.
: 포스팅 저장 서비스 - 포스팅캐시-포스팅 데이터베이스, 
  포스팅 전송 서비스 - 뉴스 피드 캐시, 
  알림 서비스
- 피드 생성
: 해당 사용자 친구들의 포스트를 시간역순으로 정렬하여 만든다.
: 포스팅 전송 서비스 - 뉴스 피드 캐시


3단계 상세 설계안

설계 대상 컴포넌트 사이의 우선순위를 정하기
- 특정 시스템 컴포넌트들의 세부사항을 깊이 있게 설명하기
- 여러분이 그 해시 함수의 설계를 구체적으로 설명하는 걸 원한다.
부정적 신호 : 불필요한 세부사항 알고리즘에 대래 시간을 쓰지 말라

ex> 포스팅 전송 서비스
1. 친구 id 목록 추천
2. 친구 데이터 추출
3. 메시지 큐
4. 포스팅 전송 작업 서버
5. 뉴스피드 캐시

이 단계에 오신 여러분이 달성한 상태
- 시스템에서 전반적으로 달성해야 할 목표와 기능 범위 확인
- 전체 설계의 개략적 청사진 마련
- 해당 청사진에 대한 면접관의 의견 청취
- 상세 설계에서 집중해야 할 영역들 확인


4단계 마무리
- 설계 결과물에 관련된 몇 가지 후속 질문
- 스스로 추가 논의를 진행하도록 할 수 있다.
- 비판적 사고 능력을 보이고, 마지막으로 좋은 인상을 남길 기회

다음의 몇가지 지침
- 시스템 병목구간 or 좀 더 개선 가능한 지점을 찾아내라.
- 여러분이 만든 설계를 다시 한번 요약해주는 것
- 오류 발생시 무슨일이 생기는 지 따져보기
- 미래에 닥칠 규모 확장 요구에 어떻게 대처할 것인지
- 필요하지만 다루지 못햇던 세부적 개선사항들



해야 할 것
- 질문을 통해 확인하라
- 문제의 요구사항을 이해하라
- 요구사항을 정확하게 이해했는지 확인
- 여러 해법을 함께 제시하라
- 개략적 설계에 면접관이 동의하면 컴포넌트의 세부사항을 설명하기
- 면접관의 아이디어를 이끌기
- 포기하지 말라

하지 말아야 할 것
- 전형적인 면접 문제들을 대비 하지 않는 자세
- 요구사항이나 가정들을 분명히 하지 않은 상태
- 처음부터 서부사항을 너무 깊이 설명하는 자세
- 진행 중에 막혔다면, 힌트를 정하지 않는 자세
- 소통을 주저하는 자세
- 설계안을 내놓는 순간 면접이 끝난 상태
